# MaCode 重构手记（备忘）

> **性质**：个人向开发备忘；**事实级版本差异**以 `CHANGELOG.md` 为准，**方向**见 `docs/roadmap.md`，`docs/progress.md` 为弃用保留的**过程档案**。本文件仅记**动机、弯路与习惯**；细节以仓库与提交记录为准，情节允许带一点事后整理。  
> **读者**：未来的作者自己，以及需要理解「为什么 Harness 会长成这样」的协作者。

---

## 附录 A：从 MVP 到 v0.9.0 的一段平实记录

### 一、为什么想要做一个 MaCode Harness

我想要的并不是「又一个动画库封装」。当时更朴素的需求是：**让数学动画这条链路对人和 Agent 都可见 (尤其对 coding agent)**——渲染、缓存、拼接、压缩，最好是文件系统里能 `ls` 出来的东西，而不是黑盒里的内存流。UNIX 管道、小工具组合、文本契约，'Bash is all you need'的口号令我振奋。对我这种写代码全凭vibe，一个 latex 公式重叠都得让 coding agent 反复搓好几轮代码的人来说，其实是在给自己写一个工具————一个能够让 coding agent 自主。

### 二、v0.5.0 前后 MVP 成型的那种兴奋

当第一批场景真能从零走到 `final.mp4`，当 `manifest.json` 真能切换引擎口径，当人第一次觉得「我也可以像教程里那样产出一小段东西」——那种兴奋很具体：**不是架构优雅，而是“它能结束”**。MVP 的快感往往来自闭环：我写完了，它跑完了，文件名对得上预期。那一瞬间很容易产生一种错觉：**路线已经选对，剩下的只是填空**。

### 三、无意识里写的「上帝脚本」

兴奋过后，很自然就开始往一个地方塞逻辑：校验、门禁、超时、缓存、编码、分发，全塞进「顺手的」入口。美其名曰编排，实际上是**让一个文件看起来像什么都知道**。我并没有故意作恶，只是觉得「这次先写在一起，以后要拆」。但「以后」在 vibe coding 里常常等于「永不」。

还有一类上帝行为更隐蔽：**入口脚本 multiplying**。今天加一个 `serve.mjs`，明天加一个 `playwright-render.mjs`，后天再加 guardian、pool——每个都有道理，合在一起却是**没有人能一次性在脑子里跑通的全链路**。某次复盘时翻到旧文档里还写着 `upload.sh`、`deliver.sh` 之类的示例，我才尴尬地发现：**大脑里跑的是新版本，手指头记住的是旧命令**。

### 四、Harness 2.0 如何反过来约束我

Harness 2.0 对我而言，不是一种「更酷的名字」，而是一组自我约束：**编排和执行要拆开，生命周期要可查，状态最好能落在文本里**。当我被迫把「启动谁、等谁、杀谁」从即兴 bash 搬进更明确的边界时，痛点是立刻的：**以前一行能糊过去的地方，现在要起文件名、要写清楚输入输出**。但回报也实在：后来者（包括两个月的我）少说一点谎。

### 五、沉入高级功能：复杂编排、MC、composite

功能一旦跑通，人很容易爱上难题。Motion Canvas、浏览器、并行、composite、shader 管线——每件事单独看都像「硬核提升」。我确实有过那种沉浸感：**调试到凌晨，只为看一帧像素统计对不对**，或为省几百兆内存去研究 Browser Pool。

composite 曾一度承载了我对「工业感」的想象：多分轨、拼接、音频、参数注入……直到默认路由统一到 `composite-unified`、legacy 必须用环境变量才敢打开时，我才慢慢接受：**高级的不仅是拓扑，更是对默认路径的勇气——简单的那条路必须最真**。

### 六、v0.9.0：大规模重构与「还债」的选择

拖到 v0.9.0 附近做重构，不是因为我突然懂架构了，更像是**欠债利息已经贵过新功能**。技术债包括但不限于：超时叠两层谁先死、默认值在 YAML / CLI / 文档里谁说了算、SOURCEMAP 的 TODO 与实际不同步、「能跑的路径」多于「敢写进 README 的路径」。

文档债则更显形：例如在修 README、`AGENTS.md` 时才注意到——

- **`concat.sh` 两种模式**：视频拼接时第一个参数是输出文件；我曾经按直觉把参数顺序写反，看起来像一行小错，实际是**让读者复制即失败**。  
- **`deliver.py` 三个位置参数**：不是「传场景目录了事」；我脑子里还留着已经不存在的 shell 封装名。  
- **Git Hooks**：hook 保护的不仅是 `pipeline/`，还有根目录的 `project.yaml`、`requirements.txt`、`package.json`；文档若只写三个目录，会给人一种「我根本没改禁区」却仍被拒绝的荒诞感——**是我在防自己**。  
- **日志文件名**：并不存在一个永远叫 `last_run.log` 的文件；真正能依赖的是 `.agent/log/` 下那一串带时间戳的名字。对着空气 tail，只会 tail 到自己的幻觉。  
- **push 卡住**：明明本地提交叠好了，远端却像在另一个宇宙；网络和 KEX、WSL、SSH，任何一层不配合，都会把那天的成就感削成一句「算了明天再说」。

这些琐碎瞬间单独看都小，合在一起却是一个教训：**体感上的「完工」常常是局部的**。

`progress.md` 头顶后来加了一行弃用提示、内容却不删——这很贴合我当时的心态：**不是否定过去那段记录，而是承认它不能再当罗盘**。

#### `progress.md` 为什么被标记弃用，`CHANGELOG` 为什么被采用

这件事不是「某次聊天里谁拍板换了你的文档」，而是重构工单里早有安排：`docs/refactor-todo.md` 的 **S0-4（文档死锁解锁）** 要求——在 `progress.md` 顶部加弃用语、**正文不删**，同时建立 **`CHANGELOG.md`** 与 **`docs/roadmap.md`**。**采用 CHANGELOG，是重构阶段在仓库里逐步落地下来的约定，不是一篇备忘能否决或替代的事。**

两种文件的用处本来就不是同一类：

- **`progress.md`** 更像**开发手记**：会话完成项、POC、评估、一长串待办掺在一起，信息量高，也**容易和代码、和别的文档不同步**。继续把它当「权威导航」，就又会回到「四处各写一版」的认知税里（重构里已经吃过「默认 engine 几处说法不一」的苦）。
- **`CHANGELOG.md`** 更像**发布级变更摘要**：按版本或可发布的块记「做了什么、对外有什么影响」，给人的问题是 **diff 口径的事实**，不是用来重读一整套心路历程。

`progress.md` **没有被清空**，相当于**档案**：实验怎么想的、链路怎么评估的，仍然有查阅价值；只是从角色上改称：**别再拿它当日历式的真源**。以后若只记得一句分工，可以是——**史料在 progress（档案），敢说「信了不会害你」的版本差异在 CHANGELOG；方向感在 roadmap**。

### 七、愿意保留的 vibe coding 习惯（好的那种）

1. **先让最小闭环落地**：只要能稳定产出可检查的中间态，就不必先争论终极形态。  
2. **真源要少而硬**：引擎契约、默认值、黑名单，宁可承认「只有这一处说了算」，也不要四处复制同一句谎言。  
3. **删掉比加上更需要勇气**：冗余入口、`preview`、旧 composite 分支，能迁到 experimental 就别留在主CLI里当诱饵。  
4. **还债要预约**：changelog、路线图、工单池，本质是未来的自己给现在的自己发短信。  
5. **把小瞬间写下来的价值**：hooks 拦了谁、`--no-verify` 何时合理、哪个环境变量是逃生门——**这些不适合靠默契**。

### 八、如何警醒自己：少在无意识里堆屎山

我会用几句很土的话反复问自己：

- **「这是第几个入口？」** 若新增命令不能一句话说清和旧入口的差异，多半是重复。  
- **「三个月后我还会记得吗？」** 若只记得 vibe、不记得接口，就把它写进可 grep 的真源（YAML、changelog、skills），别写进聊天框。  
- **「默认值一致吗？」** `project.yaml`、CLI fallback、示例 manifest、文档里的 engine，哪怕一行不一致，都会在未来变成信任和时间的税。  
- **「谁在说谎：代码还是文档？」** 当以代码为准修了文档的那一刻，其实是在还利息；若以文档为准去改代码，要先问是不是文档落后。  

屎山往往不是一天堆成的，是一天又一天**『先这样以后再整理』**。Harness 对我而言，一半是工具，一半是**对着镜子说「你以后也会骗自己」**。若这份附录还能提醒未来的我：在推进 v1.0 或下一个大功能之前，先看一眼hook、入口处、changelog——那它就值得留在 `docs/` 里占这一小块硬盘。
